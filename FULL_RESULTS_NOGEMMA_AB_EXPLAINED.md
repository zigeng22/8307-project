## 3. 项目研究问题与最初预期

### 3.1 初始研究问题

我们最初希望验证以下直觉：

1. Base 是最低基线
2. Fine-tuned 相比 Base 会提升
3. Base+RAG 相比 Base 会提升
4. Fine-tuned+RAG 应该最好，因为同时拥有领域适配和外部知识增强

这是一种合理的初始假设，也是 Proposal 阶段最自然的叙事。

### 3.2 为什么这个预期没有完全实现

真实实验中，多任务场景往往不会出现“所有任务都沿同一方向提升”的简单模式，原因包括：

1. Fine-tuning 优先提升训练分布附近的能力
2. RAG 优先提升知识依赖任务
3. 短文本分类和长文本生成对上下文噪声的敏感度不同
4. 微调后的语言偏好与检索上下文不一定天然协同

因此，出现“Task2 提升明显、Task3 更依赖 RAG、Task1 反而下降”的现象，在 NLP 实证研究里是正常的。

## 5. 评估协议：我们到底在测什么

这是当前文档最重要的一节，因为它决定你们的实验是否“可评估且有意义”。

### 5.1 为什么不能只用一个指标

因为三个任务本质不同：

1. Task1 是分类任务，关注“类别判对没有”
2. Task2 是开放生成任务，关注“回答是否像参考回答、是否语义接近”
3. Task3 是问答任务，关注“答案中是否包含正确关键信息”

单一指标无法同时覆盖这三类任务，因此需要按任务选指标。

### 5.2 Task1：为什么用 Accuracy 和 Macro-F1

#### Accuracy 是什么

Accuracy 表示：

- 全部样本里，有多少比例被模型判对了类别。

它反映的是模型的总体正确率，是最直观的分类指标。

#### 为什么不能只看 Accuracy

因为心理健康分类通常不是完全均衡的。如果某些类别样本更少，模型即使对少数类表现很差，也可能因为多数类表现尚可而保持较高 Accuracy。

#### Macro-F1 是什么

Macro-F1 会：

1. 对每个类别单独计算 F1
2. 再对所有类别做简单平均

这样每个类别权重相同，不会因为某一类样本多就主导整体指标。

#### 为什么要用 Macro-F1

因为心理健康任务里，某些类别虽然样本少，但风险高，比如：

1. Suicidal
2. Bipolar
3. Personality Disorder

如果模型只擅长 Normal/Stress 这类常见类，却对高风险类别区分不好，单看 Accuracy 会掩盖问题。Macro-F1 能更好反映模型是否对所有类别都“公平地有效”。

#### Task1 指标反映什么

1. Accuracy 高：整体分类正确率高
2. Macro-F1 高：对各类别都较均衡，不只是偏向常见类
3. Accuracy 高但 Macro-F1 低：通常意味着模型偏向头部类别，对少数类区分差
4. Accuracy 和 Macro-F1 同时下降：说明增强方法对整体分类都不利

#### 我们当前实现里还有什么

实现上，Task1 还会计算 per-class F1，用于后续错误分析。虽然它不在主表中展示，但非常适合放到附录或误差分析章节。

### 5.3 Task2：为什么用 ROUGE-L 和 BERTScore

Task2 是心理咨询对话生成。这里最大问题是：

- 同一个意思可能有很多种正确表达。

如果只用严格匹配，几乎所有生成模型都会看起来“错了”，这显然不合理。

#### ROUGE-L 是什么

ROUGE-L 基于最长公共子序列，关注预测文本和参考文本在词序和内容覆盖上的重合程度。

它更偏“表层重合”：

1. 词语是否相近
2. 内容结构是否接近
3. 关键内容是否被覆盖到

#### 为什么用 ROUGE-L

因为心理咨询回复虽然允许表达变化，但仍然需要：

1. 覆盖核心信息
2. 回应用户问题
3. 包含一些关键咨询语义内容

ROUGE-L 能较好衡量“模型有没有说到点子上”。

#### BERTScore 是什么

BERTScore 用上下文化语义向量比较预测与参考，不要求完全同词同句。

它更偏“语义相似”：

1. 即使措辞不同，只要意思接近，也能拿到较高分
2. 对 paraphrase 更宽容

#### 为什么 Task2 要同时用 ROUGE-L 和 BERTScore

因为对话生成只看 ROUGE-L 会过度惩罚“换一种说法但意思对了”的情况；只看 BERTScore 又可能忽略内容结构与关键细节覆盖。

两者结合能覆盖：

1. 表层内容重合程度
2. 深层语义接近程度

#### Task2 指标反映什么

1. ROUGE-L 提升：回答更贴近参考回复的内容组织和覆盖范围
2. BERTScore 提升：回答语义更接近参考答案，即使措辞不同也更合理
3. ROUGE-L 提升但 BERTScore 不升：可能只是表面更像，语义收益有限
4. BERTScore 提升但 ROUGE-L 不明显：可能说明模型学会了更自然的同义表达

#### 为什么 Task2 的自动指标仍有局限

因为心理咨询回复还涉及：

1. 共情程度
2. 专业性
3. 安全性
4. 是否鼓励继续表达

这些很难由自动指标完整捕捉。所以如果报告篇幅允许，建议补充少量人工案例分析。

### 5.4 Task3：为什么用 token F1、Exact Match、ROUGE-L

Task3 是医疗问答任务。问答和对话不同，评价重点不是“语气是否自然”，而是“答案是否正确地说到了关键知识点”。

#### token F1 是什么

token F1 看预测答案和参考答案的 token 重叠程度，允许部分匹配得分。

这很重要，因为医学问答里：

1. 回答可能长短不同
2. 有时只要包含关键事实，就说明模型基本答对了
3. 完全一致字符串匹配太苛刻

#### 为什么 token F1 很关键

它是最能体现“知识点是否对上”的指标。对生成式 QA 来说，token F1 比 Exact Match 更实用。

#### Exact Match 是什么

Exact Match 要求预测答案与参考答案几乎完全一致（当前实现中做了 strip 和小写归一化）。

#### 为什么还要保留 Exact Match

因为它提供一个“最严格正确率”视角：

1. 若 EM 高，说明模型经常能给出标准化、精确答案
2. 若 EM 很低但 token F1 不低，说明模型经常答到关键点，但表达方式不同或不够标准化

#### 为什么 Task3 还要看 ROUGE-L

因为很多医学回答不是超短 factoid，而是包含多句解释。ROUGE-L 可以补充看：

1. 长回答是否覆盖参考答案的结构信息
2. 生成内容是否较完整

#### Task3 指标反映什么

1. token F1 高：关键医学信息覆盖更好
2. EM 高：答案更规范、更精确
3. ROUGE-L 高：长回答结构与内容更接近参考答案
4. token F1 高但 EM 接近 0：模型知道“答什么”，但不一定按参考格式表达，这在生成式 QA 中很常见

#### 为什么我们当前 EM 大多接近 0 不是灾难

因为你们当前任务是生成式问答，不是抽取式问答。生成模型很少逐字逐句和参考答案完全一致，所以：

1. EM 可以保留，但不应作为唯一核心结论指标
2. token F1 与 ROUGE-L 更有解释力

### 5.5 这套指标为什么符合课程要求

课程要求强调：

1. 项目必须可评估
2. 不能只展示挑选出的好例子
3. 需要有严谨的分析计划

我们的做法满足这点，因为：

1. 每个任务都有与任务性质匹配的指标
2. 使用自动化批量评估，而不是只看几个样例
3. 同时考虑严格指标与宽松指标，避免单指标误导
4. 后续还能自然衔接错误分析与案例分析

## 6. 数据、方法与结果之间的关系

### 6.1 现在的 Fine-tuning 为什么更偏向 Task2

当前微调训练主要基于 MentalChat 对话数据，因此它优先优化的是：

1. 对话风格
2. 回复连贯性
3. 咨询式表达

这天然更接近 Task2，而不是 Task1 分类或 Task3 QA。

### 6.2 为什么 RAG 更容易帮助 Task3

RAG 的本质是：

1. 检索相关医学知识
2. 再把知识拼进 prompt
3. 让模型参考这些知识回答

这对知识型问答天然更有帮助，但对短文本分类未必有利，因为：

1. 分类不一定需要外部知识
2. 额外长上下文可能稀释短文本里的标签信号

### 6.3 为什么 Fine-tuned+RAG 不一定最好

这是因为它并不是“两个好东西简单相加”。

潜在冲突包括：

1. 微调后模型形成了更强的生成偏好
2. 检索上下文加入后，模型注意力被分散
3. prompt 长度上升，噪声增加
4. 对话风格偏好与事实知识检索不一定一致

所以出现非叠加现象，是合理的研究发现，不是实现错误。

## 9. 主实验结果怎么读：任务层面的解释

### 9.1 Task1：分类任务

Task1 上，多数 RAG 配置都下降，这说明：

1. 对短文本分类来说，外部检索上下文不是天然有益
2. 检索文本可能引入噪声和注意力分散
3. 分类任务更需要紧凑、判别性强的输入，而不是更长的上下文

当前最强的 Task1 配置是：

- Mistral Large baseline = 0.6036

这说明 API 大模型在零样本分类上的先验能力较强。

### 9.2 Task2：对话生成任务

Task2 上，Fine-tuned 明显优于 baseline，特别是：

1. Qwen finetuned：ROUGE-L 0.2366
2. Llama finetuned：ROUGE-L 0.2317

说明：

1. LoRA 的确学到了咨询对话风格与内容模式
2. 训练分布和评估分布在 Task2 上高度对齐

这里可以合理得出结论：

- Fine-tuning 对领域对话生成是有效的。

### 9.3 Task3：医疗问答任务

Task3 上，RAG 效果最明显：

1. Qwen base_rag：0.3070
2. Llama base_rag：0.2846
3. Mistral base_rag：0.3182

说明：

1. 外部知识对问答最关键
2. 模型并不是“不会回答”，而是“缺少足够知识支撑”
3. 一旦补入检索内容，QA 表现会稳定提升

当前 Task3 最强配置是：

- Mistral Large base_rag = 0.3182

## 10. 为什么主实验没有得到“统一最优”

这是整份文档的核心解释。

### 10.1 不是因为项目白做了

如果项目真的失败，通常会出现：

1. 结果不稳定、重复跑差异很大
2. 每个模型结果都随机波动，找不到规律
3. 不同配置之间没有任何可解释结构

而你们现在恰恰相反：

1. 结果有明确结构
2. 不同任务上有稳定分化规律
3. 后续 B1/B2 还能复现实验并做参数提升

所以这不是“无结果”，而是“结果与最初直觉不同”。

### 10.2 更可能的真实结论是什么

真正合理的结论不是：

- 一种增强方法可以在所有任务上统一最优

而是：

- Fine-tuning 更适合 Task2 这类风格/表达依赖任务
- RAG 更适合 Task3 这类知识依赖任务
- Task1 不适合当前 RAG 设计
- Fine-tuned+RAG 并非天然叠加，需要更强的任务对齐或参数对齐

这个结论比“所有配置都往上升”更有研究价值，因为它揭示了方法与任务之间的匹配关系。

## 11. AB 对齐实验：为什么要做

主实验结束后，团队最担心的是：

1. 结果是不是偶然波动？
2. 现象是不是代码 bug 导致？
3. 有没有低成本方法让结果更接近我们的预期？

因此追加 AB 实验：

1. A 组：当前主实验结果，作为 control
2. B 组：小范围对齐实验，作为 treatment

目标不是推翻主实验，而是验证：

1. 结果是否稳定可复现
2. 是否存在低成本的协议优化空间

---

## 12. B1：可复现性与路由复核实验

### 12.1 B1 做了什么

B1 不是为了“提分”，而是为了验证主结果是否稳定。

我们重跑了两个最关键的路由点：

1. Task1 使用 baseline
2. Task3 使用 base_rag

模型选择：

1. Qwen
2. Mistral

这样既覆盖开源模型，也覆盖 API 模型。

### 12.2 B1 结果

| 模型 | 指标 | A组 | B1 | Delta |
|---|---|---:|---:|---:|
| qwen2.5-7b | task1_accuracy | 0.5614 | 0.5604 | -0.0010 |
| qwen2.5-7b | task3_token_f1 | 0.3070 | 0.3070 | +0.0000 |
| mistral-large | task1_accuracy | 0.6036 | 0.6026 | -0.0010 |
| mistral-large | task3_token_f1 | 0.3182 | 0.3161 | -0.0021 |

### 12.3 B1 说明了什么

1. 这些 delta 非常小，基本都在可忽略波动范围内
2. 说明主实验不是“跑偏一次”的偶然结果
3. 说明当前 pipeline 没有明显实现错误
4. 说明任务分化是稳定现象，而不是随机波动

B1 的价值在于：

- 它为后续调参实验提供了可信的对照基线。

## 13. B2：Task3 top-k 检索强度消融

### 13.1 为什么做 B2

既然 RAG 在 Task3 已经明显有效，我们进一步问：

- 现在的 top-k=3 真的是最优吗？

如果不是，那么主实验里 RAG 没有更强提升，可能只是因为检索强度没有调对，而不是方法本身没用。

### 13.2 B2 做了什么

在 Task3 上比较：

1. top-k = 1
2. top-k = 3（control）
3. top-k = 5

模型：

1. Qwen2.5-7B
2. Mistral Large

### 13.3 B2 结果

| 模型 | k=1 | k=3（Control） | k=5 | 最优k | 相对k=3提升 |
|---|---:|---:|---:|---:|---:|
| qwen2.5-7b | 0.2863 | 0.3070 | 0.3225 | 5 | +0.0155 |
| mistral-large | 0.2947 | 0.3182 | 0.3366 | 5 | +0.0184 |

### 13.4 B2 结果说明了什么

1. top-k=1 太保守，信息不够
2. top-k=3 有提升，但仍非最优
3. top-k=5 在两个模型上都更好

这说明：

1. Task3 上 RAG 不是没效果，而是还可以通过参数对齐进一步变强
2. 原主实验中的 top-k=3 是一个合理起点，但不是当前最佳点
3. “RAG 的收益不稳定”并不完全是方法问题，也可能是参数选择问题

## 14. 到目前为止，最稳妥的总结是什么

### 14.1 你们已经证明了三件事

1. 主实验现象是真实的，不是偶然
2. Fine-tuning 与 RAG 的收益高度任务依赖
3. 至少在 Task3 上，参数对齐能继续提升 RAG 效果

### 14.2 所以项目应怎么讲

不要继续坚持：

- Base 最低，Fine-tuned 更高，Base+RAG 更高，Fine-tuned+RAG 最高

因为这不符合你们的真实数据。

更好的写法是：

- 在心理健康多任务场景中，不同增强方法并不存在统一最优配置。Fine-tuning 更适合对话生成，RAG 更适合知识型问答，而分类任务在当前检索设计下不一定受益。进一步的 B1/B2 对齐实验表明，这种现象是稳定且可优化的，其中 Task3 在将 top-k 从 3 提高到 5 后可获得进一步增益。

这是一种更强、更可信、也更符合课程要求的结论。

## 15. 可以直接写进 Final Report 的指标分析段落（中文版）

在评估设置上，我们针对三类任务采用了任务匹配的指标体系。对于 Task1 分类任务，我们使用 Accuracy 和 Macro-F1，其中 Accuracy 衡量总体正确率，而 Macro-F1 对每个心理状态类别赋予相同权重，能够更公平地反映模型在少数类和高风险类别上的表现。对于 Task2 对话生成任务，我们同时使用 ROUGE-L 和 BERTScore，其中 ROUGE-L 衡量生成回复与参考回复在内容覆盖和结构上的重合程度，BERTScore 则衡量语义层面的相似度，能够部分缓解生成任务中“表达不同但语义正确”的问题。对于 Task3 医疗问答任务，我们使用 token-level F1、Exact Match 和 ROUGE-L，其中 token F1 用于评估关键医学信息覆盖程度，Exact Match 提供严格正确率视角，ROUGE-L 则补充衡量较长回答与参考答案在结构和内容上的接近程度。我们采用多指标而非单一指标，是因为三个任务的输出形式与评价重点差异明显，这样的设计也更符合课程项目对于“系统性评估而非少量示例展示”的要求。

## 16. 可以直接写进 Final Report 的结果讨论段落（中文版）

主实验结果并未支持“单一增强策略在所有任务上统一提升”的假设，而是呈现出显著的任务依赖性。Fine-tuning 在对话生成任务上表现出最明确的收益，例如 Qwen 与 Llama 在 Task2 上均有显著提升；相比之下，RAG 在医疗问答任务上效果更突出，如 Qwen、Llama 和 Mistral 在 Task3 上均取得比 baseline 更高的 token-level F1。与此同时，Task1 分类任务在加入检索上下文后普遍出现下降，说明对短文本分类而言，额外的检索内容可能带来噪声而非帮助。进一步地，Fine-tuned+RAG 并未稳定优于 Fine-tuned 或 Base+RAG，表明参数高效微调与检索增强之间存在非线性交互，而非简单可加。后续 B1 可复现性实验验证了主结果的稳定性，B2 检索强度消融实验则进一步表明，Task3 的 RAG 表现对检索深度敏感，在 qwen 和 mistral 上将 top-k 从 3 调整到 5 后均获得进一步提升。因此，本项目的核心发现并非“增强方法失效”，而是“增强方法有效但高度任务依赖，且其效果受到任务-方法匹配与参数对齐的显著影响”。

## 19. 目前最推荐的最终结论

### 19.1 科学上最稳的结论

在心理健康多任务场景下，Fine-tuning 与 RAG 并不存在统一最优配置。Fine-tuning 更适合对话生成任务，RAG 更适合知识问答任务，而分类任务在当前检索设计下未必受益。主实验现象经过 B1 可复现性实验验证，且通过 B2 检索强度消融进一步证明，Task3 的 RAG 表现仍有可观的参数对齐提升空间。

### 19.2 工程上最实用的结论

如果把项目看成一个实际系统：

1. Task1：优先使用 Mistral Large baseline
2. Task2：优先使用 Qwen finetuned
3. Task3：优先使用 Mistral base_rag，并考虑 top-k=5

也就是说，按任务路由配置比坚持单一全局配置更有效。
